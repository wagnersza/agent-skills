# Custom Operators

Production-ready patterns for building Kubernetes operators with CRDs and the Operator SDK.

## CRD Definition

CustomResourceDefinition with schema validation for a Database custom resource.

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: databases.app.example.com
spec:
  group: app.example.com
  names:
    kind: Database
    listKind: DatabaseList
    plural: databases
    singular: database
    shortNames: [db]
  scope: Namespaced
  versions:
    - name: v1
      served: true
      storage: true
      subresources:
        status: {}
      additionalPrinterColumns:
        - { name: Engine, type: string, jsonPath: .spec.engine }
        - { name: Replicas, type: integer, jsonPath: .spec.replicas }
        - { name: Status, type: string, jsonPath: .status.phase }
        - { name: Age, type: date, jsonPath: .metadata.creationTimestamp }
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              required: [engine, storage, replicas]
              properties:
                engine:
                  type: string
                  enum: [postgres, mysql, redis]
                storage:
                  type: string
                  pattern: "^[0-9]+(Gi|Ti)$"
                replicas:
                  type: integer
                  minimum: 1
                  maximum: 7
                version:
                  type: string
                  default: "16"
            status:
              type: object
              properties:
                phase:
                  type: string
                  enum: [Pending, Creating, Running, Failed]
                readyReplicas: { type: integer }
                conditions:
                  type: array
                  items:
                    type: object
                    properties:
                      type: { type: string }
                      status: { type: string }
                      lastTransitionTime: { type: string, format: date-time }
                      reason: { type: string }
                      message: { type: string }
```

## Custom Resource Instance

Create a Database resource using the CRD defined above.

```yaml
apiVersion: app.example.com/v1
kind: Database
metadata:
  name: orders-db
  namespace: production
spec:
  engine: postgres
  version: "16"
  storage: 50Gi
  replicas: 3
```

## API Types (Go)

Spec and Status structs with kubebuilder validation markers.

```go
// +kubebuilder:object:root=true
// +kubebuilder:subresource:status
// +kubebuilder:printcolumn:name="Engine",type=string,JSONPath=`.spec.engine`
// +kubebuilder:printcolumn:name="Replicas",type=integer,JSONPath=`.spec.replicas`
// +kubebuilder:printcolumn:name="Phase",type=string,JSONPath=`.status.phase`

type Database struct {
    metav1.TypeMeta   `json:",inline"`
    metav1.ObjectMeta `json:"metadata,omitempty"`
    Spec              DatabaseSpec   `json:"spec,omitempty"`
    Status            DatabaseStatus `json:"status,omitempty"`
}

type DatabaseSpec struct {
    // +kubebuilder:validation:Enum=postgres;mysql;redis
    Engine string `json:"engine"`

    // +kubebuilder:validation:Pattern=`^[0-9]+(Gi|Ti)$`
    Storage string `json:"storage"`

    // +kubebuilder:validation:Minimum=1
    // +kubebuilder:validation:Maximum=7
    // +kubebuilder:default=1
    Replicas int32 `json:"replicas"`

    // +kubebuilder:default="16"
    Version string `json:"version,omitempty"`
}

type DatabaseStatus struct {
    Phase         string             `json:"phase,omitempty"`
    ReadyReplicas int32             `json:"readyReplicas,omitempty"`
    Conditions    []metav1.Condition `json:"conditions,omitempty"`
}
```

## Reconciler Pattern (Go)

Fetch the CR, check finalizers, create or update dependent resources, and update status.

```go
func (r *DatabaseReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    // 1. Fetch the CR
    var db appv1.Database
    if err := r.Get(ctx, req.NamespacedName, &db); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }

    // 2. Check finalizer for cleanup on deletion
    finalizerName := "app.example.com/finalizer"
    if db.DeletionTimestamp != nil {
        if controllerutil.ContainsFinalizer(&db, finalizerName) {
            if err := r.cleanupResources(ctx, &db); err != nil {
                return ctrl.Result{}, err
            }
            controllerutil.RemoveFinalizer(&db, finalizerName)
            return ctrl.Result{}, r.Update(ctx, &db)
        }
        return ctrl.Result{}, nil
    }
    if !controllerutil.ContainsFinalizer(&db, finalizerName) {
        controllerutil.AddFinalizer(&db, finalizerName)
        if err := r.Update(ctx, &db); err != nil {
            return ctrl.Result{}, err
        }
    }

    // 3. Create or update dependent resources
    sts := r.buildStatefulSet(&db)
    if err := controllerutil.SetControllerReference(&db, sts, r.Scheme); err != nil {
        return ctrl.Result{}, err
    }
    if _, err := controllerutil.CreateOrUpdate(ctx, r.Client, sts, func() error {
        r.mutateStatefulSet(sts, &db)
        return nil
    }); err != nil {
        return ctrl.Result{}, err
    }

    // Repeat for Service, PVC, ConfigMap as needed...

    // 4. Update status
    db.Status.Phase = "Running"
    db.Status.ReadyReplicas = sts.Status.ReadyReplicas
    meta.SetStatusCondition(&db.Status.Conditions, metav1.Condition{
        Type:               "Available",
        Status:             metav1.ConditionTrue,
        Reason:             "ReconcileSuccess",
        Message:            "All dependents are ready",
        LastTransitionTime: metav1.Now(),
    })
    return ctrl.Result{}, r.Status().Update(ctx, &db)
}

func (r *DatabaseReconciler) SetupWithManager(mgr ctrl.Manager) error {
    return ctrl.NewControllerManagedBy(mgr).
        For(&appv1.Database{}).
        Owns(&appsv1.StatefulSet{}).
        Owns(&corev1.Service{}).
        Complete(r)
}
```

## RBAC for Operators

ServiceAccount, ClusterRole, and ClusterRoleBinding for CRD and dependent resource access.

```yaml
apiVersion: v1
kind: ServiceAccount
metadata: { name: database-operator, namespace: database-system }
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: database-operator
rules:
  - apiGroups: [app.example.com]
    resources: [databases, databases/status, databases/finalizers]
    verbs: [get, list, watch, create, update, patch, delete]
  - apiGroups: [apps]
    resources: [statefulsets]
    verbs: [get, list, watch, create, update, patch, delete]
  - apiGroups: [""]
    resources: [services, persistentvolumeclaims, configmaps, secrets]
    verbs: [get, list, watch, create, update, patch, delete]
  - apiGroups: [""]
    resources: [events]
    verbs: [create, patch]
  - apiGroups: [coordination.k8s.io]
    resources: [leases]
    verbs: [get, list, watch, create, update, patch, delete]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata: { name: database-operator }
roleRef: { apiGroup: rbac.authorization.k8s.io, kind: ClusterRole, name: database-operator }
subjects:
  - { kind: ServiceAccount, name: database-operator, namespace: database-system }
```

## Operator Deployment

Deploy the operator with leader election and resource limits for high availability.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: database-operator
  namespace: database-system
  labels: { app: database-operator }
spec:
  replicas: 2
  selector:
    matchLabels: { app: database-operator }
  template:
    metadata:
      labels: { app: database-operator }
    spec:
      serviceAccountName: database-operator
      containers:
        - name: manager
          image: example.com/database-operator:v1.0.0
          args: [--leader-elect, --health-probe-bind-address=:8081, --metrics-bind-address=:8443]
          ports:
            - { containerPort: 8081, name: health }
            - { containerPort: 8443, name: metrics }
          livenessProbe: { httpGet: { path: /healthz, port: 8081 }, initialDelaySeconds: 15, periodSeconds: 20 }
          readinessProbe: { httpGet: { path: /readyz, port: 8081 }, initialDelaySeconds: 5, periodSeconds: 10 }
          resources:
            limits: { cpu: 500m, memory: 256Mi }
            requests: { cpu: 100m, memory: 64Mi }
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            runAsNonRoot: true
            capabilities: { drop: [ALL] }
```

## Best Practices

1. Make reconciliation idempotent — use `CreateOrUpdate` with mutate functions so reprocessing the same CR produces identical dependent resources.
2. Use status conditions (`metav1.Condition`) to communicate operator state; follow the standard `type`, `status`, `reason`, `message` convention.
3. Always register a finalizer when the operator creates external resources (cloud infrastructure, DNS records) that need cleanup on CR deletion.
4. Enable leader election when running multiple operator replicas to prevent concurrent reconciliation of the same resource.
5. Set owner references on all dependent resources so Kubernetes garbage collection automatically cleans up when the parent CR is deleted.
6. Test reconciliation logic with `envtest` (controller-runtime's test framework) to run against a real API server without a full cluster.
7. Version CRD schemas carefully — use conversion webhooks when introducing breaking changes, and serve multiple versions during migration periods.
